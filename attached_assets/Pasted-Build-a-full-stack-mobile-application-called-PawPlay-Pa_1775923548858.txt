Build a full-stack mobile application called PawPlay.

PawPlay is a gamified dog training app for pawrents (pet owners,
not professional dog trainers). The app turns daily dog training
into a game for the whole family using randomised command
sequences, scoring, streaks, and family leaderboards.

SCORING ENGINE
Implement as a pure utility function: calculateScore()
═══════════════════════════════════════════════════════════════

Session total = 100 participation points
Commands per session = 5
Base points per command = 20

COMPLY WINDOWS BY DIFFICULTY:
  easy   = 6 seconds    completion bonus = 0
  medium = 4 seconds    completion bonus = +10
  expert = 2 seconds    completion bonus = +25

SCORING RULES PER COMMAND:
  Complete within window        = +20 pts
  Exceed window (no skip)       = -1 pt per second over
  Skip (no exceed)              = -20 pts
  Skip after exceeding window   = -20 pts AND -1 pt per
                                  second already accumulated

SCORE FLOORS:
  easy   = display floor 0 (never show negative to user)
  medium = display floor 0 (never show negative to user)
  expert = no floor (negatives shown in red)

Always store the TRUE raw score in the database.
Apply the floor only at the display layer.

BONUS EVALUATION (run after all 5 commands complete):
  perfect_round   all 5 within window + 0 skips → +20 pts
  speed_demon     all under half the window      → +5 per cmd
  combo_streak    3+ consecutive within window   → x1.5 on those
  clean_sweep     0 skips total                  → +10 pts
  first_cue       no resets on any command        → +3 per cmd
  difficulty_bonus medium → +10 | expert → +25
  family_record   beats leaderboard top score     → crown badge
                  + push notification to all family members

BONUS CAP: total bonus points cannot exceed +50 per session

PARTICIPATION POINTS (what user sees):
  easy + medium = max(0, raw_score + bonuses)
  expert        = raw_score + bonuses (can be negative)

COMMAND LEVEL SYSTEM
Run evaluateCommandLevel() after every session is saved
═══════════════════════════════════════════════════════════════

Level 1 — Learning
  Condition : command exists in library
  Obedience gate : NO

Level 2 — Practising
  Condition : training_sessions_count >= 5
  Obedience gate : NO

Level 3 — Reliable
  Condition : qb_successes_count >= 10
              AND qb_sessions_with_success >= 3
  Obedience gate : YES

When a command levels up:
  - Update level in Replit Database
  - Trigger level-up animation on profile command chip
  - Send push notification: "Sit just reached Reliable! 🎉"

OBEDIENCE CHALLENGE GATE:
  Count commands at Level 3 for this dog
  If count >= 7 → challenge is unlocked
  Progress ring shows: "X of 7 commands Reliable"

BASIC COMMAND LIST (v1 — 7 commands):
  Sit, Down, Stay, Come, Heel, Place, Leave it

CRITICAL IMPLEMENTATION NOTES
═══════════════════════════════════════════════════════════════

1. The HOLD button timer must use Reanimated 2 shared values
   running on the native thread. Do NOT use setTimeout or
   React setState for the countdown. This is non-negotiable —
   timer accuracy is core to fair scoring.

2. The demo session (before login) must be entirely local state.
   Zero API calls. Zero database writes. Nothing is saved.

3. Raw score is always stored in the database.
   Score floor (0 for easy/medium) is a display-only rule.
   Apply it only when rendering the score to the user.

4. evaluateCommandLevel() must run server-side after every
   POST /api/sessions. The app refetches commands after
   saving a session to pick up any level changes.

5. Fredoka One must be fully loaded before the Challenge Active
   screen renders the command word. Show a loading state if
   the font is not ready — do not fall back to system font.

6. Family invite uses a 6-character alphanumeric code generated
   on family creation. Share via the invite code in Settings.

7. Variable reward schedule default is OFF for all new users.
   The tooltip is always visible below the toggle whether
   it is on or off.

8. Always On must be enabled in Replit Deployments so the
   backend never sleeps between mobile API calls.
